### План тестирования страницы «Школа 21» (Корпоративное обучение)

| № | Пункт плана | Описание |
|--|--|--|
| **1** | **Объект тестирования** | Веб-страница «Групповые интенсивы для руководителей и команд»: `https://21-school.ru/corporate-training/gruppovye-intensivy-dlia-rukovoditelei-i-komand` |
| **2** | **Цель тестирования** | Проверка корректности отображения (UI), адаптивности, кросс-браузерной совместимости и сквозной функциональности интерактивных элементов страницы для обеспечения эталонного пользовательского опыта. |
| **3** | **Область тестирования** | - **Контент:** заголовки, описания программ, списки навыков, FAQ, проверка текстовых блоков на отсутствие орфографических ошибок;<br>- **Навигационные элементы:** внутренние ссылки, кнопки перехода, внешние редиректы (соцсети, политика конфиденциальности);<br>- **Интерактивные формы:** валидация полей (граничные значения, маски ввода), отправка заявки на обучение, подписка на обратную связь;<br>- **Отправка данных:** проверка того, что при нажатии на кнопку отправки формы данные уходят без ошибок и сайт получает правильный ответ от сервера через вкладку Network;<br>- **Графические элементы:** изображения, иконки, баннеры (качество, скорость загрузки, корректность пропорций);<br>- **Адаптивность и UI:** корректность отображения на различных разрешениях экранов (Desktop, Tablet, Mobile) с учетом масштабирования интерфейса (вплоть до 500%). |
| **4** | **Виды тестирования** | - Функциональное тестирование;<br>- Тестирование пользовательского интерфейса (UI);<br>- Тестирование адаптивности и мобильное тестирование;<br>- Кросс-браузерное тестирование;<br>- Тестирование форм и валидации (позитивные/негативные сценарии). |
| **5** | **Критерии начала тестирования** | - Страница развернута на тестовом окружении и доступна по указанному URL;<br>- Предоставлены актуальные дизайн-макеты (Figma) и требования к контенту;<br>- Подготовлены тестовые данные (валидные/инвалидные email, номера телефонов). |
| **6** | **Критерии окончания тестирования** | - Все запланированные проверки из чек-листа и тест-кейсов выполнены;<br>- Блокирующие, критические и высокоприоритетные дефекты полностью устранены и перепроверены;<br>- Формы обратной связи успешно отправляют данные на уровне фронтенда и API-запросов;<br>- Отображение страницы соответствует макетам в основных целевых браузерах и ОС. |
| **7** | **Ресурсы и окружение (Environment)** | - **Аппаратное обеспечение:** Рабочая станция QA (ПК/Ноутбук), реальные мобильные устройства (iOS, Android);<br>- **Операционные системы:** Windows, macOS, Ubuntu, iOS, Android;<br>- **Браузеры (Десктоп):** Google Chrome, Mozilla Firefox, Microsoft Edge, Safari, Яндекс Браузер;<br>- **Браузеры (Мобильные):** Mobile Safari, Chrome Mobile;<br>- **Инструменты:** Chrome DevTools (вкладки Console, Network, Elements), Figma;<br>- **Доступ к сети интернет.**|
| **8** | **Тестовая документация** | - Чек-лист для экспресс-проверки контента, UI и кросс-браузерности;<br>- Тест-кейсы для детальной проверки функциональности, валидации полей и отправки форм. |
| **9** | **Ориентировочные сроки** | - **Подготовка тестовой документации:** 2.5–3 часа;<br>- **Выполнение тестирования:** 5–6 часов (включая кросс-браузерные/кросс-платформенные проверки);<br>- **Подготовка отчета о тестировании:** 1 час. |
| **10** | **Критерии приостановки и возобновления** | **Приостановка:** обнаружение блокирующих дефектов (страница возвращает 5xx/4xx ошибки;  полная неработоспособность форм отправки заявок);<br>- **Возобновление:** после устранения блокирующих дефектов и предоставления исправленной версии с подтверждением от команды разработки. |
| **11** | **Риски и митигация** | - **Риск:** Ограниченные временные ресурсы могут сократить объем кросс-браузерного тестирования;<br>- **Риск:** Изменение контента или макетов в процессе тестирования потребует регрессионных проверок;<br>- **Риск:** Нет доступа к внутренней панели сайта или CRM, поэтому нельзя напрямую увидеть, доходят ли заявки пользователей до системы.|
